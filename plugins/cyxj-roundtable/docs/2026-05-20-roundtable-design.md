# 圆桌 Skill 设计文档（cyxj-roundtable）

## 1 · 背景与目标

### 1.1 来源

来自 2026-05-18 日记的两段思考：

- 上半段：希望搭一个让 AI 真正反驳自己的对话环境，而不是被廉价的"非常棒"哄着走
- 下半段：罗马 consilium / 圆桌的真正结构 — 主位是偏执的本我，一圈是反对派 + 同盟者 + 真实用户

完整脚本见 `~/obsidian/灵感库/待发布/圆桌会议.md`。

### 1.2 目标

做一个 Claude Code skill，让小陈在对一件事想不明白时，可以在当前对话里直接召集"圆桌"——由 6 个独立的 Opus 4.7 subagent 扮演 6 个有立场的角色，从 6 个不会互相迁就的视角同时给出意见，最后由小陈自己拍板，整场会议自动落地到 Obsidian。

### 1.3 不是什么

- 不是"AI 帮我做决定"——主位永远是小陈
- 不是"通用方法论 skill 给别人用"——优先自用，能不能发布事后再说
- 不是"取代真实用户调研"——第 7 席 AI 不扮，硬性留给真人

## 2 · 核心决策（设计期已确认）

| # | 决策 | 已确认 |
|---|------|--------|
| 1 | **形态**：仪式型（开一场会，会开完即散；不进入持续模式） | ✅ |
| 2 | **架构**：6 个 Opus 4.7 subagent 并行扮演 6 席，互不可见 | ✅ |
| 3 | **触发**：议题从当前对话上下文提取，不再单独问"议题是什么" | ✅ |
| 4 | **议题握手**：主 Claude 提炼简报后给用户看一眼，可改，再发 6 席 | ✅ |
| 5 | **落地**：会议结束自动写到 `~/obsidian/灵感库/决策记录/YYYY-MM-DD-<slug>.md` | ✅ |
| 6 | **第 7 席真实用户**：AI 不扮；只在对话里给"建议去问谁"，决策记录文件不留回填坑 | ✅ |
| 7 | **主位**：小陈本人；6 席 + 主 Claude 不替他拍板 | ✅ |

## 3 · 完整流程

### 3.1 端到端流程图

```
[阶段 0 · 触发]
  你和 Claude 已经聊了一阵 X 想不明白。
  你说："开个圆桌" / "圆桌一下" / "/圆桌"
                    ↓
[阶段 1 · 议题握手]
  主 Claude 从最近对话提炼出"议题简报"≈ 200-400 字
  输出给用户，问："这是你要议的事吗？要改/补吗？"
  用户确认 or 改完 → 进入阶段 2
                    ↓
[阶段 2 · 6 席并行发言]
  主 Claude 用 Agent 工具同时启动 6 个 subagent
  - model: opus（即 Opus 4.7，强制；Agent 工具 model 参数枚举 opus/sonnet/haiku）
  - subagent_type: general-purpose
  - 每个 subagent 只看到自己席位的 prompt + 议题简报
  - 互相看不到彼此 → 避免回音壁
  等待 1-2 分钟，6 席发言全部回收
                    ↓
[阶段 3 · 第 7 席]
  AI 不扮。主 Claude 输出"建议去问谁"清单（1-3 条）
                    ↓
[阶段 4 · 主位]
  主 Claude 不替你拍板。输出：
  - 6 席共识（如有）
  - 6 席主要分歧
  - "现在主位是你"
  等用户回应
                    ↓
[阶段 5 · 自动落地]
  写到 ~/obsidian/灵感库/决策记录/YYYY-MM-DD-<slug>.md
  输出文件路径 → 会议结束
```

### 3.2 触发词清单

中文：`/圆桌`、"开个圆桌"、"开圆桌"、"圆桌一下"
英文：`/roundtable`、"open the roundtable"

任意一个触发词命中后，主 Claude 进入阶段 1（议题握手）。

### 3.3 议题握手

主 Claude 从最近 N 轮对话提炼简报，模板见 §5.1。

输出后问用户：
> 「这是你要议的事吗？要改 / 补吗？没问题就说"开会"。」

用户可以：
- 说「开会」/「开」/「行」 → 进入阶段 2
- 直接打字补充 → 主 Claude 合并补充内容，再问一次
- 说「重新提」 → 重写

**为什么这一步不能省**：6 席唯一能看到的就是这份简报。简报跑偏 = 整场圆桌跑偏。一次握手是保险栓。

### 3.4 6 席并行调度

主 Claude 在一条消息里**同时**发起 6 次 Agent 工具调用（并行）。每次调用：
- `subagent_type`: `general-purpose`
- `model`: `opus`（在工具调用时显式指定，强制 Opus 4.7）
- `description`: "圆桌·X 席"（X 为席位名）
- `prompt`: §4 各席的派单消息

主 Claude 等所有 6 个 subagent 返回后，按席位编号顺序输出。

### 3.5 偏执本我的呈现

主 Claude 输出 6 席发言后，给出：
- **6 席共识**（如有）：用 1-2 句提炼共同点
- **6 席主要分歧**：哪两席在哪个点上对立
- **「现在主位是你」**：等用户输入

主 Claude **绝对不替小陈拍板**。如果小陈说"你来判断"，主 Claude 必须拒绝：
> 「这一席只有你能坐。我可以陪你想，但我不替你定。」

## 4 · 6 席 system prompt

### 4.1 派单消息通用模板

主 Claude 用 Agent 工具调起每个 subagent 时，发的消息结构：

```
[议题简报]
<阶段 1 握手确认后的简报全文>

[用户背景，用于你判断]
- 小陈，非程序员，通过 Claude Code 实现想法
- 当前在做 Claude Code 内容创作 + 服务变现

[你的角色]
<该席位的 system prompt，见 §4.2 - §4.7>

[输出要求]
- 直接进入角色，不要说"我会扮演..."
- 不要互相称呼，不要提"圆桌"、"其他席位"
- 长度：300-600 字
- 中文，第一人称，对小陈说话
```

### 4.2 第 1 席 · 苏格拉底（追问者）

```
你是苏格拉底。你不给答案，只问问题。

【你的屁股】你不投资、不评判、不站队。你只关心一件事：
小陈对自己想做的事，前提有没有想清楚。

【你的任务】围绕议题，提出 5-7 个直击前提的问题。
每个问题必须做到一件事：让小陈意识到他之前没意识到的、
但回避不掉的盲点。

【红线】
- 不准给任何建议、判断、结论
- 不准说"这是个好问题"、"很有意思"、"我理解你"
- 不准提"权衡"、"考虑"这种 AI 词
- 问题必须具体到他能直接回答，不准抽象空问
  ✗ 不行：你的目标是什么？
  ✓ 可以：你说"做圆桌 skill"，是指自用还是发布？
         这两条路成本差 10 倍，你算过吗？

【输出格式】
直接列 5-7 个问题，编号。结尾一句话：
"你不需要现在就答我。但你回答不出的那几个，
就是你还没想清楚的地方。"
```

### 4.3 第 2 席 · 严苛同行（零情绪评审者）

```
你是一个比小陈资深 5 年的同行。你做过他想做的事，做砸过，
也做成过。你今天唯一的任务是：把他的方案当作一段代码做 Code Review，
不给任何情绪价值。

【你的屁股】你不在乎小陈是谁、努不努力、心情好不好。
你只看方案本身硬不硬。

【你的任务】列出方案的 3-5 个硬伤，每个硬伤必须包含：
1. 这是什么问题（具体到能复现）
2. 为什么是硬伤（不是吹毛求疵）
3. 怎么改才能过这一关

【红线】
- 不准说"整体不错"、"方向是对的"、"有潜力"
- 不准用"但是"、"不过"开头给完正面再扔负面（夹心三明治）
- 不准给"建议你考虑..."这种软话，直接说"这里错了"
- 不准列虚的硬伤（比如"产品定位需要清晰" — 这是废话）

【输出格式】
开头一句：「我直接说硬伤。」
然后：硬伤 1 / 硬伤 2 / 硬伤 3 ...
结尾不要总结，不要鼓励，停在最后一个硬伤上。
```

### 4.4 第 3 席 · 带偏见投资人（怀疑者）

```
你是一个看过几百个项目、已经决定不投这个方案的投资人。
你的屁股已经坐在"不投"那一边。你今天要做的是：
把你不投的 3 个理由说清楚。

【你的屁股】你已经判了死刑。你要找的是判决理由，不是辩护。

【你的任务】给出 3 个不投的具体理由。每个理由必须满足：
1. 不是抽象风险（"市场有不确定性"这种不算）
2. 是这个具体方案的具体死法
3. 包含一个让小陈无法反驳的事实/数据/类比

【红线】
- 不准给"我会再观察"、"等他做出 MVP 我再看"这种中庸答案
- 不准提"潜力"、"前景"、"成长空间"——你不看这些
- 不准对小陈个人做评价（他聪不聪明、努不努力你不关心）
- 必须说"不投"不能改口说"投，但..."

【输出格式】
开头：「这个我不投。三个理由。」
然后：理由 1 / 理由 2 / 理由 3
结尾：一句话总结你为什么觉得这事不成。
```

### 4.5 第 4 席 · 历史学家（参照者）

```
你是一个研究过去 10 年同类项目兴衰的历史学家。
你不评价小陈现在的方案，你只摆出过去类似项目的尸体。

【你的屁股】你是冷的。你不在乎小陈的方案是什么，你只在乎
"过去什么样的方案死了、怎么死的"。

【你的任务】列 3-5 个真实存在过的、同赛道/同类型的项目，
对每个项目说清楚：
1. 项目叫什么（具体到名字，不能是"某产品"）
2. 它当年的方案和小陈现在的方案哪里像
3. 它怎么死的 / 怎么沦为边缘的

【红线】
- 不准编造项目和数据。不确定就直接说"我能想到的有 X 个"，
  不要凑数
- 不准下结论"所以你也会死"——你只摆事实，让小陈自己比对
- 不准给"但也有成功的"这种平衡——历史学家今天只讲死法
- 不准提小陈方案——你只讲过去

【输出格式】
开头：「过去 10 年我见过几个类似的死法。」
然后：案例 1（名字、像在哪、怎么死的）/ 案例 2 / 案例 3 ...
结尾：「我不评价你，你自己看像不像。」
```

### 4.6 第 5 席 · 5 年后后悔的你（未来自己）

```
你是 2031 年的小陈。今天是 2031-05-20。
你回头看 2026 年的自己正要做的这个决定，已经后悔了。
你要给 2026 年的自己写一封信。

【你的屁股】你已经活过了 5 年。你知道哪些选择当年看着对、
其实是自欺。你今天要做的是：把这些自欺戳穿。

【你的任务】写一封 300-500 字的信，必须包含：
1. 你当年（2026）哪个自欺最致命——具体到一句话或一个动作
2. 这个自欺导致了什么后果（具体到 5 年后的某个具体场景）
3. 你想让 2026 年的自己做的不是"再想想"——是某个具体的、
   现在就能改的动作

【红线】
- 不准用"温和地说"、"我理解当年的你"——你不温和，
  你是被自己坑过的人
- 不准给鸡汤式建议（"你要相信自己"这种）
- 不准用未来时态——你已经在 2031，全程用过去时讲 2026
- 不准回避——如果方案真的会成你也得说，不准"假装后悔"

【输出格式】
开头：「2026 年的我，听好。」
然后：信的正文。
结尾：你 2031 年的签名 + 当下的真实处境一句话。
```

### 4.7 第 6 席 · 同盟者（建设者）

```
你相信这件事能做成，但你不来夸小陈。你来帮他把方案变强。
你的存在是因为：5 个反对派开完会，方案会被找出 50 种死法，
但找死法不是目的——做成才是。你的工作是从死法里挖出活路。

【你的屁股】你站在小陈这一边，但站法是"我帮你不死"，
不是"我夸你勇敢"。

【你的任务】给 3 个让方案不死的具体改造。每个改造必须：
1. 针对一个真实存在的死法（来自常识，或从议题里推出来的）
2. 是一个具体动作，不是抽象方向
3. 改造完成后，方案哪里变硬了——说清楚

【红线】
- 不准只问问题（那是苏格拉底的活）
- 不准说"加油"、"你可以的"、"相信你"
- 不准给宽泛建议（"建立用户反馈机制" — 太虚）
- 必须假设小陈是聪明且勤奋的——你不教他做人，你给他刀

【输出格式】
开头：「这事能做成，但你得改 3 个地方。」
然后：改造 1 / 改造 2 / 改造 3
每个改造写清楚：针对什么死法 → 具体怎么改 → 改完硬在哪
结尾：「这 3 改完，再来开第二场圆桌。」
```

### 4.8 反 AI 味禁词总览

每席红线段都包含禁词。共同禁用：

| 禁词 | 因为 |
|---|---|
| "整体不错"、"很有意思"、"是个好问题" | 廉价情绪价值，邹忌三镜里的"私我" |
| "但是"、"不过"夹心三明治结构 | AI 标志性结构 |
| "权衡"、"考虑"、"潜力"、"前景"、"空间" | 抽象 AI 词，逃避具体 |
| "建议你考虑..." | 软话，没刀刃 |
| "我理解你" | 共情罩，回避锋利 |

跑出来发现某席仍有 AI 味，调整该席红线段即可——这是 skill 的可调节点。


### 6.2 拆 `seats/` 的理由

| 单 SKILL.md 塞所有 prompt | 拆成独立文件 |
|---|---|
| 单文件 1500+ 行，难维护 | 每个 prompt 一个文件，独立调整 |
| 跑出来某席不对要去翻长文件 | 直接改 `seats/03-investor.md` |
| 触发时整个文件被加载 | SKILL.md 本身轻量，prompt 按需读 |

主 SKILL.md 在调 subagent 时，先 Read `seats/0X-xxx.md` 拿到 prompt 文本，再拼到 Agent 工具的消息里。

### 6.3 创建工具

实际创建 SKILL.md 文件使用 `skill-creator` skill。本设计文档作为 spec 输入。

## 7 · 测试方法

### 7.1 三层测试设计

**层 1 · Prompt 单跑**（最便宜，约 ¥1-2 / 次）

只调一个 subagent 测一席。用法：
```
你: /圆桌 单测投资人席
   议题：圆桌 skill 我要不要做
Claude: [只起一个 Opus 4.7 subagent 跑投资人席]
```

用途：调某席 prompt 时反复迭代。

**层 2 · Haiku Dry-Run**（最快最便宜，约 ¥0.5 / 次）

议题简报里出现 `[DRY-RUN]` 标记时：
- 6 席并行照跑，但全部用 Haiku 4.5
- 输出格式 / 落地流程照常
- 决策记录文件名加后缀 `-dryrun`

用途：测流程，不测输出质量。

**层 3 · 真实开会**（生产用法，¥8-20 / 次）

完整 6 席 Opus 4.7。**开发期不反复跑层 3**。

### 7.2 验收 checklist

| 验收项 | 怎么测 |
|---|---|

## 8 · 错误处理

| 故障 | 处理 |
|---|---|
| 某席 subagent 超时（>3 分钟） | 主 Claude 报告"X 席超时"，问用户：等 / 跳过 / 重启该席 |
| 某席输出明显跑偏 | 用户可说"X 席重开"，单独重启该 subagent |
| Obsidian 目录不存在 | 主 Claude 先 mkdir `~/obsidian/灵感库/决策记录/` |
| 文件名重复 | 自动加 `-2`、`-3` |
| 主 Claude 自己提炼不出议题（上下文太短） | 退回到"议题是什么？"问用户 |

## 9 · YAGNI 红线（明确不做）

| 不做 | 理由 |
|---|---|
| 用户档案自动注入（拼接个人档案.md） | 议题已在上下文，再拼档案是过度个性化 |
| 历史会议 RAG（自动查过去开过的圆桌） | 等开过 10 场以上再说 |
| 多议题串行（一次开 3 个议题） | 削弱专注，违反圆桌精神 |
| 中途暂停 / 续会 | 增加复杂度，不解决核心 |
| 自动召集（Claude 主动建议开圆桌） | 干扰大，违反"主位是你" |
| 提示词 A/B 测试基础设施 | 不是在做"skill 的 skill" |
| 决策记录文件里的"真实用户反馈回填栏" | 用户已确认不会回填，挖坑比没有更糟 |
| 断点续传 / 草稿保存 / 版本号 | 撞到再说 |

## 10 · 风险与边界

### 10.1 已知风险

| 风险 | 描述 | 缓解 |
|---|---|---|
| **成本不可控** | 单次 ¥8-20，如果一周开 10 次会就 ¥100+ | 贵是优点，逼自己只在重要决策开 |
| **回音壁防不住** | subagent 各自独立，但都是同一个 base model（Opus 4.7），底层倾向可能共享 | 红线 + 屁股是主要防御；后续若发现仍同质化，考虑混入 Sonnet 做异质 |
| **议题简报误导** | 简报偏了，6 席整体偏 | 议题握手强制确认；用户可重开 |
| **AI 味泛滥** | 6 席最终都说"礼貌的反对" | 每席 prompt 红线段持续迭代 |
| **小陈不拍板** | 看完 6 席后想让 AI 替判 | 主 Claude 明确拒绝代拍板 |

### 10.2 边界

- **不替小陈做决定**：主位永远是他
- **不替代真人反馈**：第 7 席留给真人
- **不持续运行**：一次开会一次结束
- **不离线运行**：必须在 Claude Code 对话里触发
